**Why Laravel 13 requires it:** the package previously capped `illuminate/database` at `^12.0` and Testbench at `^10.0`, and pinned `turso/libsql` to the unstable `dev-master`. Composer will not install Laravel 13 (which ships with Testbench 11) under those caps.

```diff

-    "turso/libsql": "dev-master",

+    "turso/libsql": "^0.2.5",

-    "orchestra/testbench": "^9.9||^10.0"
```

- `illuminate/database: ^11.0|^12.0|^13.0` — adds L13, **keeps L11/L12**.

### 3.2 `src/Database/LibsqlConnection.php` — `select()` signature

**Why Laravel 13 requires it:** Laravel 13 added a 4th parameter to `Illuminate\Database\Connection::select()`:

```php
// vendor — Laravel 13
public function select($query, $bindings = [], $useReadPdo = true, array $fetchUsing = [])
```

PHP requires a child override to remain signature-compatible with its parent. Because `LibsqlConnection extends Connection`, the missing `$fetchUsing` parameter would otherwise trigger a *"Declaration must be compatible"* fatal at class-load time.

**Original:**
```php
public function select($query, $bindings = [], $useReadPdo = true)
{
```

**Replacement:**
```php
public function select($query, $bindings = [], $useReadPdo = true, array $fetchUsing = [])
{
```

The new parameter is accepted to satisfy the signature contract. The libsql driver returns rows via its own fetch-mode `match()` block, so `$fetchUsing` (a Laravel-level row-hydration callback) is intentionally not consumed — behavior is unchanged on L11/L12, where the extra optional parameter is simply never passed.

> Note: an earlier in-progress edit had the correct signature but with stray blank lines between each parameter; this was normalized to a single clean line (PSR-12).

### 3.3 `src/Database/LibsqlSchemaGrammar.php` — `compileVectorIndex()` override

**Why Laravel 13 requires it:** Laravel 13 promoted vector indexes to a **native** schema feature:

- `Illuminate\Database\Schema\Blueprint::vectorIndex($column, $name = null)` is now a real method.
- It registers a `vectorIndex` command compiled by `Illuminate\Database\Schema\Grammars\Grammar::compileVectorIndex()`, whose default body **throws** `RuntimeException('The database driver in use does not support vector indexes.')`.

This package previously provided vector-index support through a `Blueprint::macro('vectorIndex', ...)` (in `src/Vector/VectorMacro.php`). On L11/L12 that macro is the only `vectorIndex` implementation, so it runs. **On L13 the native method shadows the macro** (real methods are resolved before `__call`/macros), so calls route to the throwing base `compileVectorIndex()` — surfacing as the `VectorDataTest` failures:

```
RuntimeException: The database driver in use does not support vector indexes.
  at vendor/laravel/framework/.../Grammars/Grammar.php:168
```

**Fix — added override (no original to replace; this is new):**
```php
use Illuminate\Database\Schema\Blueprint;
// ...
public function compileVectorIndex(Blueprint $blueprint, Fluent $command): string
{
    return sprintf(
        'create index %s on %s (libsql_vector_idx(%s))',
        $command->index,
        $blueprint->getTable(),
        implode(', ', (array) $command->columns)
    );
}
```

Design notes:
- Emits the **same SQL shape** as the original macro (`CREATE INDEX <name> ON <table>(libsql_vector_idx(<column>))`), so the `nearest()` query macro keeps working — it references the index by its exact string name inside `vector_top_k('<name>', ...)`.
- Uses **raw, unwrapped** identifiers deliberately. This grammar overrides `wrap()` to replace `"` with `'`, so `wrapTable('movies')` would yield the string literal `'movies'` and produce invalid DDL. Raw names mirror the legacy macro and the columns access matches the parent `SQLiteGrammar::compileIndex()` (`$command->index` / `$command->columns`).
- The `VectorMacro::create()` macro is **retained** for backward compatibility with Laravel 11/12, where the native `Blueprint::vectorIndex()` does not exist. On L13 it is harmlessly shadowed.

## 6. Remaining risks

1. **Vector index unwrapped identifiers.** `compileVectorIndex()` does not wrap the table/column names (intentional — `wrap()` would break the DDL). Index/table/column names containing spaces or reserved words would not be quoted. This matches the original macro's behavior, so it is not a regression, but it is a latent edge case.
2. **Hardcoded, expired test credentials.** `tests/TestCase.php` pins `http://127.0.0.1:8081` and a JWT whose `exp` is in early 2025. The bundled `richan/turso-dev` image accepts it locally, but this is brittle and unrelated to framework compatibility.
3. **`shouldSkipTests()` always returns `false`** (`tests/Pest.php`), so the network suites never self-skip — they hard-fail without a running server rather than being marked skipped. Consider gating on an env var if CI parity is a concern.
4. **Duplicate/legacy class trees.** `src/LibsqlConnection.php` and `src/LibsqlSchemaGrammar.php` (root `Libsql\Laravel` namespace) shadow the active `Libsql\Laravel\Database\*` classes and are not wired into the service provider. They currently load fine on L13 but are dead code that could drift; worth removing in a follow-up.
5. **PHPStan not configured.** No `phpstan.neon` exists, so "static analysis passes if configured" is satisfied vacuously. Running ad-hoc (`--level=5`) surfaces only pre-existing `Fluent` magic-property false-positives (the same pattern Laravel's own grammars use) — none originate from the changed files.
6. **PHP 8.5 RC.** Local runs used PHP 8.5.0RC5. No version-specific issues observed, but production targets should pin a stable PHP.
